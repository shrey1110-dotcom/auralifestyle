PROPOSED SOLUTION

Problem Addressed

  • Fragmented channels causing slow resolution.
  • Manual triage + inconsistent priorities.
  • Limited self-service; repetitive user queries.
  • Poor visibility & delayed stakeholder alerts.

Solution Highlights

  • Unified API handles Web, Chat, Email, GLPI/Solman.
  • Automatic classification & routing with deterministic rules.
  • Historical learning adjusts low-confidence routing.
  • Instant KB article suggestions (TF‑IDF similarity).

TECHNICAL APPROACH

Architecture Diagram

                  [KB Engine]<--+                   (Email/SMS/Webhooks)

Stack

  • Backend: FastAPI + SQLAlchemy + SQLite (→ Postgres).
  • NLP: Rule-based classification + confidence scoring.
  • KB: TF‑IDF vector search (scikit-learn).
  • Frontend: React (Vite) + Service Worker + Manifest.
  • Notifications: SMTP (smtp4dev), optional SMS (Twilio), Discord/Telegram.
  • Adapters: GLPI / Solman mapping to unified model.

Methodology (Phases)

  • MVP → Ingestion + Classification + KB + Alerts.
  • Enhancement → Auto-assign + Historical Prior + PWA.
  • Scaling → Postgres + Metrics + Dashboard.
  • Advanced → RBAC, embedding classifier, SLA timers.

FEASIBILITY & VIABILITY

Feasibility

  • Open-source stack; low infra cost; local/offline friendly.
  • Deterministic classification ensures transparency.
  • Adapters simplify legacy tool convergence.

Challenges & Mitigation

  • Uncommon issues misclassified → fallback “ServiceDesk” + human override.
  • Alert overload → configurable ALERT_EVENTS + severity filters.
  • SMS constraints (DLT) → multi-channel (Email/Telegram) backup.
  • Security & PII → planned RBAC & audit logs.

Scalability Path

  • Horizontal API scaling via containers.
  • Upgrade to embeddings for large KB sets.
  • Add Observability (Prometheus/Grafana).
  • Multi-tenancy & role-based partitioning.
